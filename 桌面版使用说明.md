# 财务算薪系统桌面版使用说明

这套系统现在可以走本地桌面工具路线，不需要部署到公网服务器，也不需要让别人连你电脑上的浏览器开发地址。

## 适合什么场景

- 你自己在本机使用
- 财务同事在各自电脑本地使用
- 暂时不想买云服务器
- 希望继续沿用现在这套前端 + 后端 + Excel 计算逻辑

## 当前实现方式

桌面版本质上是给现有项目加了一层本地应用壳：

- Electron 打开一个本地桌面窗口
- 自动拉起本地后端
- 本地网关统一提供页面和接口
- 页面里继续走现有上传、计算、导出流程

也就是说：

- 对用户来说，它像一个桌面软件
- 对项目来说，底层还是你现在这套成熟的算薪逻辑

## 现在已经新增的本地兜底能力

为了让桌面版更接近“解压即用”，现在后端已经补了一层本地兜底：

- 如果本地 MySQL 可用，仍然优先使用 MySQL
- 如果 MySQL 不可用，上传记录会自动退回到本地 JSON 文件存储
- 桌面版的上传文件、流程状态也已经开始写入应用自己的本地数据目录，而不是项目源码目录
- 桌面版启动时会自动生成自己的运行配置文件，不再完全依赖源码根目录下的 `.env`

这意味着：

- 桌面版不再因为“数据库没启动”就完全无法工作
- 至少上传记录、流程状态这类轻量数据已经可以本地化
- 用户的数据和程序文件开始分离，更适合打包、分发和长期使用
- Redis 也不是当前这套桌面链路的必需依赖
- 桌面 bundle 也已经开始只保留运行真正需要的服务端依赖，不再原样夹带整套开发依赖

## 如何判断当前机器跑的是哪种模式

现在健康接口除了 `ok` 之外，还会返回存储模式信息。

你可以看：

- `storage.uploadDir`
- `storage.uploadRecordMode`

其中：

- `database` 表示当前走数据库
- `local-json-fallback` 表示当前已经退回本地 JSON 兜底

## 已完成的关键验证

已经验证过一件非常重要的事：

- 当数据库不可连接时，后端会自动退回本地 JSON 模式
- 本地上传记录文件会自动创建到应用数据目录下
- 在这种模式下，健康接口和上传记录列表接口都可以正常返回
- 在这种模式下，Excel 上传接口也可以正常工作，上传文件、上传记录、流程状态文件都会写入本地目录
- 在这种模式下，绩效确认、工资草稿生成、绩效结果导出、绩效回填导出、工资草稿导出、工资回填导出也已经验证可用

这说明桌面版已经不再是“数据库一挂就完全不可用”的状态。

## 一键回归验证命令

现在已经补了一条一键冒烟命令，用来验证桌面本地 fallback 模式：

```bash
cd /Users/kristen/Documents/Codex/2026-06-23/niha/finance-payroll-tool/apps/desktop
pnpm smoke:fallback
```

这条命令会自动跑通并验证：

- 数据库失联时自动退回本地 JSON
- 绩效上传
- 流程状态推进
- 绩效确认
- 工资上传
- 工资草稿生成
- 绩效导出
- 绩效回填导出
- 工资草稿导出
- 工资回填导出

但也要说明白：

- 这只是第一步
- 还不等于整个系统已经完全无环境依赖
- 如果你要做到“普通用户解压就直接用”，下一步仍建议继续做 SQLite / 本地轻量存储收口

## 和“部署服务器”相比的区别

桌面版：

- 不需要公网服务器
- 不需要域名
- 不需要别人访问你的本地 `127.0.0.1`
- 每台使用电脑本地运行一份

服务器版：

- 所有人可以通过一个网址访问
- 但需要云服务器、数据库、运维和上线部署
